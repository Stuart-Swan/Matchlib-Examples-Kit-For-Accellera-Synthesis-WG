IO Operations Involved in Scheduling Rules
------------------------------------------

There are three sets of IO operations involved in the scheduling rules:

Symbol   Description
S:       Calls to synchronization interfaces (SystemC wait, Matchlib SyncChannel calls, barriers, etc.)
R/W:     SystemC signal reads and writes
M:       Calls to message-passing interfaces (Matchlib Push/PushNB/Pop/PopNB, p2p put/nb_put/get/nb_get)

All the operations above are referred to as IO operations.


Definitions
-----------

clk(x) represents the clock cycle at which IO operation x commits.
issue(x) represents the clock cycle at which the IO operation x is issued by a process (applicable to messages).


Equivalence Rules For Pre-HLS and Post-HLS Models
-------------------------------------------------

These are the scheduling rules for IO operations that HLS must adhere to.
These rules are specified at the per-process level, but are proven to also hold at the system level.

E1 (Synchronization-order preservation).
For every process P and synchronization calls s1,s2 ∈ S_P within P:
s1 <_src s2 ⇒ clk_post(s1) < clk_post(s2)

E2 (Signal-visibility preservation).
∀ r ∈ R : clk_post(r) = clk_post(pred_S(r))
∀ w ∈ W : clk_post(w) = clk_post(succ_S(w))

E3 (Safe message issue ordering).
For every pair of message operations m1,m2 ∈ M within a single process:
(i) Same-interface source order: if m1 and m2 access the same message-passing interface 
and m1 <_src m2, then issue_post(m1) ≤ issue_post(m2).
(ii) Distinct-interface no-reverse: if m1 and m2 access distinct message-passing interfaces
and m1 <_src m2, then issue_post(m1) ≤ issue_post(m2).
Thus message operations may be issued in source order or in parallel where allowed,
but they shall not be issued in reverse source order.  

E4 (Per-channel FIFO / rendezvous semantics).
For every abstract channel c, Push_c and Pop_c actions in τ_post obey the agreed channel semantics:
messages are not dropped, duplicated, or reordered, and transfers occur only when
permitted by the channel’s bounded-FIFO or rendezvous capacity / enablement behavior.

E5 (Messages cannot cross syncs).
For every message m and synchronization call s, 
∀ m ∈ pref_S(s) : issue_post(m) ≤ clk_post(s)
∀ m ∈ suff_S(s) : issue_post(m) > clk_post(s)



Allowed Transformations During HLS
----------------------------------

T1 During loop pipelining, HLS may overlap multiple loop iterations and thereby change the
relative timing of message operations from different iterations.
The same scheduling rules still apply, with explicit pipeline-stage wait statements treated
as synchronization interface calls. HLS shall not reorder accesses to the same message-passing
interface, signal, or synchronization interface across iterations.
Automatic flush is required so that overlapped execution does not introduce
deadlocks or mismatches relative to the pre-HLS reference behavior.

T2 Pragma for direct input signals allows HLS to remove pipeline registers.
Drivers of such signals are required to hold them stable after receivers come out of reset.

T3 Pragma for direct input sync signals allows HLS to remove pipeline registers. The sync channel controls
when the driver updates such signals, and also when the pipeline ramps down and ramps up again.

T4 Array accesses remain constrained by explicit synchronization calls in the source model.
For externally visible arrays or memories, the array-access mapping layer defines how
accesses map onto external IO. Any array read or write before a synchronization call shall
commit before or in the same cycle as that synchronization call;
any array read or write after a synchronization call shall commit in a later cycle.
Internal arrays are not externally visible, so their internal scheduling is not
directly constrained except through the surrounding IO behavior.

T5 When arrays are mapped to RAMs/ROMs during HLS, accesses may be reordered if HLS can prove
that the reordering is conflict-free. T4 must still be followed.

T6 When a loop is pipelined, any signal read and write operations that occurred in the same
clock cycle in the pre-HLS model must also by default occur in the same clock cycle in the post-HLS model.
Similarly, the order of signal IO operations must be strictly preserved by default when loops are
pipelined. Pragmas/directives can be used to explicitly disable this default behavior.

T7 By default, HLS can add additional states (or latency) into a process. 
This does not affect E1-E5 as long as the system is latency-insensitive.
If the system is latency-sensitive verification techniques can be used to maintain pre-HLS
and post-HLS equivalence.

Design Rules
------------

RULE1 Signal IO should be placed adjacent to its intended synchronization anchor:
signal writes just before their associated synchronization call, and signal reads just after it.
A design is ill-formed if signal IO is separated from its intended anchor in a way that can
change what value is observed, or if the signal IO does not have a unique unconditional
synchronization anchor. A common example is a blocking message-passing call between
a signal IO operation and its intended synchronization call.

RULE2 If a rolled loop body has signal IO operations but no wait statement, the design is ill-formed.
